# blog_writer 프로젝트 현황 정리 / Codex 인수인계 문서

## 1. 프로젝트 현재 목적

이 프로젝트는 원래 여러 소스를 자동 수집해 "아스날 Today" 형태의 종합 블로그 글을 생성하는 구조였다.  
하지만 개발이 진행되면서 방향이 바뀌었고, 현재는 기사 1개를 입력받아 템플릿 1개로 블로그 포스트 1개를 만드는 구조로 재설계 중이다.

현재 목표는 아스날 관련 글을 아래 4개 템플릿 중 하나로 생성하는 것이다.

1. 인터뷰형
2. 기자회견형
3. 칼럼형
4. 매치 리포트형

즉, 예전처럼 섹션형 "오늘의 아스날" 포스트 1개를 길게 만드는 방식이 아니라, 개별 아티클 단위 포스트 발행 구조로 전환 중이다.

## 2. 핵심 기획 방향

### 유지할 것

- 대주제는 계속 아스날
- 포스트는 개별 아티클 단위로 발행
- 사람이 마지막에 HTML 검수 후 블로그 업로드
- 번역문은 전문 직역보다 블로그용 요약/재구성에 가까워야 함
- 출처는 반드시 명시
- 이미지는 가능하면 포함, 없으면 생략 가능
- 인터뷰/기자회견은 발언의 결을 살리는 것이 중요
- 칼럼은 논지, 감정선, 맥락을 살리는 것이 중요
- 매치 리포트는 경기 요약, 흐름, 결정 장면, 남은 과제를 갖춘 구조가 중요

### 피하려는 것

- Reddit 위주 자동 포스트 남발
- 영상 링크만 던져놓고 설명 없는 글
- 섹션 제목과 기사 내용이 맞지 않는 구조
- 번역투가 너무 심한 문장
- 기사 캡션, 광고, 이미지 설명이 본문에 섞이는 현상
- 아스날 블로그인데 상대팀 찬양 기사나 비관련 기사까지 섞이는 문제

## 3. 템플릿별 구현 상태

### A. 인터뷰형 템플릿

#### 목적

선수 인터뷰 기사 1개를 블로그용으로 정리한다.

#### 현재 구조

- 제목: 기사형 제목 톤
- 상단 배지: 아스날 인터뷰 정리
- 상단 이미지 포함 가능
- 본문:
- 2~4개의 요약 문단
- 중요 발언은 quote block으로 강조
- quote 앞에 한 줄 설명 문단이 들어갈 수 있음
- 마지막에 원문 제목, 출처 링크 표기

#### 대표 예시

- Declan Rice 인터뷰 기사
- `rice-looks-forward-after-wembley-defeat`

#### 반영된 보정 포인트

- 중복 문단 제거
- 제목 톤 자연화
- 인용문 앞 설명 한 줄 추가
- hero image 출력 문제 해결 완료
- `article_content_collector.py`에서 `image_url`, `article_title`, `paragraphs` 추출 강화

#### 상태 평가

상당히 만족스러운 수준이며, 실사용 가능한 단계까지 올라와 있다.

### B. 기자회견형 템플릿

#### 목적

감독 기자회견 기사에서 핵심 질문과 답변을 블로그용으로 정리한다.

#### 현재 구조

- 제목: `미켈 아르테타 기자회견 정리 - ...`
- 배지: 아스날 기자회견 정리
- 상단 이미지 포함
- 도입문 1~2개
- 질문별 섹션 라벨 + quote block 형태
- 마지막 출처 표기

#### 중요한 구현 포인트

기자회견 원문은 보통 아래처럼 질문 라벨이 이어지는 구조다.

- `on the result:`
- `on his choice of goalkeeper:`
- `on the injury:`
- `on if Ebere Eze will travel...`
- `on how he helps the players...`

따라서 템플릿에서는 질문 우선순위 선택 로직이 들어간다. 우선적으로 뽑도록 설계된 질문 범주는 아래와 같다.

- 경기 결과
- 골키퍼 선택
- 경기 흐름
- 부상
- 남은 시즌 / 반등

#### 현재 한계

- 에제 부상처럼 질문이 둘로 쪼개져 문맥이 분리되면 자연스럽게 합성되지 않는 경우가 있음
- 광고/중간 블록 때문에 질문-답변 연결이 흔들릴 수 있음
- 사람의 HTML 검수 단계에서 최종 다듬기를 열어두는 구조가 필요함

#### 상태 평가

첫 시도 대비 충분히 쓸 만하다. 다만 부상 관련 문맥 복원은 개선 여지가 남아 있다.

### C. 칼럼형 템플릿

#### 목적

The Athletic, Daily Mail 같은 칼럼을 블로그용 분석 글로 바꾼다.

#### 현재 구조

- 배지: 아스날 칼럼 정리
- 제목: 원문 논지를 살린 한국어 기사형 제목
- 상단 이미지
- 도입 1~2문단
- 필요 시 quote block
- 나머지는 문단형 본문
- 마지막에 원문 제목, 출처

#### 중요한 보정 사항

- 직역투 표현 완화
- 캡션 문구가 본문에 섞이는 문제 제거
- 논지 없는 문장 제거
- 인용문 위치 보정
- `"리그컵을 저주한다"` 같은 표현을 자연스러운 문장으로 재작성
- 카라바오컵 / 리그컵 명칭 혼용 주의
- 토니 아담스 등 발언 주체를 명시적으로 표시
- 아르테타 발언이 들어가면 화자 표시 필요
- 큰따옴표 인용은 quote block으로 묶는 편이 좋음

#### Daily Mail 대응 보정

Daily Mail 칼럼은 구조가 더 선동적이거나 수치 중심일 수 있으므로, 사카 관련 글 등에서는 아래 요소를 반영한다.

- 최근 경기력
- 수치 변화
- 상징성
- 경쟁 구도
- 아르테타 체제 내 의미
- 단순 직역이 아닌 읽히는 칼럼형 요약

#### 상태 평가

검수 전제라면 충분히 실사용 가능하다. 완성도를 과도하게 끌어올리려다 업무가 멈추기보다, 현재 수준에서 사람 검수로 마감하는 방향이 적절하다고 판단한다.

### D. 매치 리포트형 템플릿

#### 목적

경기 결과 기사를 블로그용 경기 정리 포스트로 변환한다.

#### 기본 구조

- 배지: 아스날 매치 리포트
- 제목
- 상단 이미지
- 도입문
- 섹션 예시:
- 경기 요약
- 경기 흐름
- 승부를 가른 장면
- 남은 시즌의 과제

#### 현재 이슈

- 섹션 제목과 아래 문단이 약간 어긋나는 경우가 있음
- 예를 들어 `승부를 가른 장면` 아래 문단이 경기 전체 분위기 설명처럼 보일 수 있음
- `남은 시즌의 과제` 아래 문단이 실제로는 상대팀 상황 설명인 경우도 있음

#### 상태 평가

전체 방향은 만족스럽지만, 섹션과 내용의 정합성 보정이 필요하다. 그래도 템플릿 방향 자체는 유지하기로 한 상태다.

## 4. 주요 실행 파일 역할

현재 설계상 각 템플릿은 개별 `main` 파일로 실행된다.

예상 구조:

- `main_interview.py`
- `main_presser.py`
- `main_column.py`
- `main_match_report.py`

최종적으로는 아래 파일이 추가 중심 역할을 맡게 된다.

- `main_auto.py`

이 파일은 최근 기사 수집 후 각 템플릿 후보를 자동 분류해, "오늘 올릴 만한 인터뷰 / 기자회견 / 칼럼 / 매치 리포트 후보"를 찾는 역할을 하게 하려는 상태다.

## 5. `main_auto.py`의 기획 의도

### 목적

수동으로 링크를 넣지 않아도 최근 수집 기사 중에서 4개 템플릿 후보를 자동 탐색하도록 만든다.

### 기대 동작

1. 최근 기사 수집
2. 템플릿별 후보 자동 분류
3. 예시 결과:
- 인터뷰 후보 0~5개
- 기자회견 후보 0~5개
- 칼럼 후보 0~5개
- 매치 리포트 후보 0~5개
4. 각 후보를 터미널 또는 JSON 파일로 출력
5. 사용자는 후보를 보고 직접 하나를 골라 개별 `main` 파일에 링크를 넣어 실행 가능

### 추가 요구사항

- `main_auto.py` 상단 주석에 예시 실행 명령을 남겨둘 것
- 인터뷰형 실행 코드
- 기자회견형 실행 코드
- 칼럼형 실행 코드
- 매치 리포트형 실행 코드
- 링크는 비워둬도 괜찮음

### 또 하나의 요구사항

각 템플릿별 후보 기사를 최대 5개씩 출력/저장해야 한다. 총 최대 20개까지 관리 가능해야 한다.

이유:

- 자동 추천 외에 다른 포스트로도 응용 가능
- 블로그에 여러 개 올리고 싶을 때 참고 가능

## 6. 중복 링크 관리 요구사항

### 필요성

이미 블로그에 사용한 링크를 다시 후보에 올리지 않도록 해야 한다.

### 원하는 방식

- 링크 목록을 따로 파일로 저장
- JSON이어도 되고 다른 파일이어도 상관없음
- 사용자가 수동으로 올린 링크도 직접 추가 가능
- 이 파일을 기준으로 중복 여부 체크

### 구현 의도

예시:

- `used_links.json`
- 포스트 생성 후 링크 저장
- `main_auto.py` 후보 선정 시 이미 사용된 링크는 제외

### 주의

사용자가 직접 글을 생성/업로드하는 경우가 있으므로, 이력 파일은 자동 + 수동 둘 다 관리 가능해야 한다.

## 7. 번역 정책

### 현재 합의된 우선순위

일반적으로 다음 순서로 번역한다.

1. DeepL
2. Gemini
3. LibreTranslate

### 환경변수

- `DEEPL_API_KEY`
- `GEMINI_API_KEY`
- `TRANSLATOR_PRIMARY=deepl`
- `TRANSLATOR_SECONDARY=gemini`
- `TRANSLATOR_FALLBACK=libre`
- 필요 시 `LIBRETRANSLATE_URL=http://localhost:5000/translate`

### Reddit 번역 관련

기존에는 Reddit 번역 라우터가 따로 있었고, 슬랭/밈 여부에 따라 Gemini를 우선 쓰거나 DeepL budget 확인 후 fallback 하는 구조였음.

하지만 현재 프로젝트 방향이 템플릿형 포스트 중심으로 바뀌었기 때문에, Reddit 비중은 예전보다 크게 낮아졌다.

### 현재 판단

인터뷰 / 기자회견 / 칼럼 / 매치 리포트 중심이라면 Reddit 라우팅 로직은 핵심이 아니다. 다만 기존 파일은 남아 있을 수 있으므로 정리는 필요하다.

## 8. 이미지 처리 정책

### 기본 방향

- 이미지는 가능하면 포함
- 없으면 생략
- 이미지가 있다면 아스날 중심이어야 함
- 패배 기사라도 맨시티 세리머니보다 아스날 선수나 아르테타 반응 이미지가 더 적절할 수 있음

### 현재 구현 관련 이슈

기존 섹션형 블로그 생성기에서는 `choose_section_image_candidate`가 중복 정의되어 있었음. 하지만 템플릿형 구조로 오면서 이 문제는 많이 단순화되었다.

### `article_content_collector.py`의 핵심 역할

- `article_title`
- `article_text`
- `paragraphs`
- `image_url`
- `image_caption`

기사 단위로 위 정보를 안정적으로 추출해야 한다.

### 현재 구현에서 주의할 점

- 캡션이 본문에 섞이지 않게 해야 함
- 광고/배너 이미지를 hero image로 잡지 않도록 해야 함
- trusted host / image safe 여부를 적당히 관리할 필요가 있음
- 현재 템플릿형 구조에서는 섹션별 이미지 선택보다 기사 1개에 대한 대표 이미지 선택이 더 중요함

## 9. 다음 Codex가 바로 해야 할 일

우선순위 기준으로 정리하면 아래와 같다.

1. 현재 코드베이스가 있다면 템플릿별 `main` 파일과 `article_content_collector.py`의 실제 구현 상태를 점검
2. `main_auto.py`를 설계 및 구현해 최근 기사 수집, 템플릿별 후보 분류, 최대 5개 후보 저장 구조를 만든다
3. `used_links.json` 같은 중복 링크 관리 파일을 도입해 자동/수동 링크 이력을 함께 관리한다
4. 매치 리포트형의 섹션-내용 정합성 보정 로직을 강화한다
5. 기자회견형의 문맥이 쪼개진 질문-답변 복원 품질을 개선한다
6. Reddit 관련 구형 로직은 핵심 경로에서 분리하거나 정리 여부를 판단한다

## 10. 작업 판단 기준

이 프로젝트는 "완전 자동 퍼블리싱"보다 "사람 검수 전제의 반자동 블로그 작성 도구"에 가깝다.

따라서 아래 기준을 유지하는 것이 중요하다.

- 템플릿별 결과물이 사람이 읽기에 자연스러울 것
- HTML 검수 단계에서 조금만 다듬으면 바로 올릴 수 있을 것
- 출처와 이미지가 지나치게 어색하지 않을 것
- 아스날 중심성이 흔들리지 않을 것
- 완성도 집착 때문에 전체 파이프라인 개발이 멈추지 않을 것

## 11. 한 줄 요약

`blog_writer`는 이제 "여러 소스를 한데 묶는 오늘의 아스날 생성기"가 아니라, 아스날 관련 기사 1개를 인터뷰형 / 기자회견형 / 칼럼형 / 매치 리포트형 템플릿으로 정리해 블로그 포스트 1개를 만드는 자동화 도구로 재정의된 상태다.
